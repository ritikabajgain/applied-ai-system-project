# PawPal+ Project Reflection

## 1. System Design

**a. Initial design**
The system should be allow the user to enter the general owner and pet information. It should create a general schedule for the pet care along with explaning why it chose the plan. The owner should also be able to add or edit to the schedule generated by the system.

- Briefly describe your initial UML design.
  The initial UML design consists of four classes connected by clear relationships: an Owner owns one or more Pets, each Pet has one Schedule, and each Schedule contains zero or more Tasks. The diagram captures attributes (with private visibility) and methods (with public visibility) for each class, modeling the flow from user input (owner/pet info) through task management to schedule generation.

- What classes did you include, and what responsibilities did you assign to each?
  - **Owner**: Stores the owner's name, daily available time, and care preferences. Responsible for managing preferences and updating availability, which act as constraints for scheduling.
  - **Pet**: Stores the pet's name, species, age, and a reference to its owner. Responsible for providing a summary of the pet's details to the UI and scheduler.
  - **Task**: Represents a single care activity (e.g., walk, feeding, meds) with a title, duration, priority, and category. Responsible for editing its own fields and converting its priority to a numeric value for sorting.
  - **Schedule**: The central class that ties everything together. It holds a reference to a Pet, a list of Tasks, and the owner's available time. Responsible for adding/removing tasks, generating an optimized daily plan that fits within time constraints and respects priorities, explaining the reasoning behind the plan, and reporting remaining time.

**b. Design changes**

- Did your design change during implementation?
  Yes. After reviewing the initial skeleton with AI, several missing relationships and potential issues were identified, and the design was updated before writing any logic.

- If yes, describe at least one change and why you made it.
  - **Added a `pets` list to Owner**: The original design had Pet pointing to Owner, but Owner had no way to access its Pets. Adding this list allows the system to look up all pets for a given owner.
  - **Added a `schedule` attribute to Pet**: Similarly, Pet had no link to its Schedule. Now when a Schedule is created it automatically attaches itself to the Pet, so navigating from Pet to its Schedule is straightforward.
  - **Removed the separate `available_time` parameter from Schedule**: The original Schedule stored its own copy of available time, which could get out of sync if the owner updated their availability later. Now Schedule reads it directly from the owner through the pet, so there is only one source of truth.
  - **Made `total_time` a computed property**: Instead of manually tracking total time and risking it going stale when tasks are added or removed, it is now calculated on the fly by summing task durations. This eliminates a whole category of bookkeeping bugs.
